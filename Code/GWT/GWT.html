Test Sizes
How to reduce the size of tests
- Systems can span multiple processes and machines. This doesn't mean that the tests of those systems should be complicated either. The most common technique for testing a large system with small tests is to take advantage of test doubles, which allow for fake or mock implementations of parts of the system to be used in tests. Whenever, a component of a system under test would communicate with another machine, or process, or make a non-blocking call, it should be possible to replace that component with an in-process test double. This allows the important parts of the system to be tested entirely using small tests.
Small tests are not required to use mocks extensively
- Mocks and other test doubles are a tool that make it possible to write small tests of systems with dependencies that are slow, non-deterministic, or run in different processes.Test doubles typically should not be used for other purposes when a real implementation is available that is fast and in-process. In particular, mocking out all of a class's dependencies so that the test "focuses only on a single class" usually does more harm than good.
Using Parameterized Tests
- There are some common test anti-patterns that you should avoid:
- Copying/Pasting the same test again, just with a different set of inputs
- The tests should not be in a helper method. Because then you have to jump between the helper method and the place where the actual assertion is being made in order to figure out where the error in the case of the test actually is.
- You should not have assert statements that are being called/used in while or for loops. This is because if the test fails, you wouldn't know which set of inputs actually caused the test to fail.

Flakes and Code Coverage
- Flaky tests are tests that passes or fails non-deterministically with respect to its test inputs. You should either remove such test cases itself or fix the flakiness issues.
- High code coverage does not guarantee high quality in test coverage. But a low code coverage number guarantees that large areeas are completely untested. The gains of increasing test coverage are logarithmic. While it might be worthwhile to improve the test coverage from 30% to 70%, increasing from 90% to 95%, maybe not so much.
